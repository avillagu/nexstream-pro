REGISTRO COMPLETO DE ERRORES (EN ORDEN CRONOLÓGICO)
================================================================================

ERROR #1: DOCKERFILE EN SUBCARPETA (Backend/)
─────────────────────────────────────────────
- QUÉ PASÓ: El Dockerfile fue creado dentro de Backend/. Render busca el
  Dockerfile en la raíz del repositorio por defecto.
- IMPACTO: Primer despliegue falló completamente.
  Error: "failed to read dockerfile: open Dockerfile: no such file or directory"
- CAUSA RAÍZ: El Agente Backend no tenía en su prompt instrucciones sobre la
  estructura del repositorio en relación con el CI/CD de Render. El Senior no
  validó la ubicación del Dockerfile antes de aprobar.
- PREVENCIÓN: Todo Dockerfile debe ir en la RAÍZ del repositorio. Incluir en
  el prompt del agente: "El Dockerfile debe estar en el directorio raíz del
  proyecto, no dentro de subcarpetas."

ERROR #2: RUTAS COPY DEL DOCKERFILE INCORRECTAS TRAS MOVERLO
─────────────────────────────────────────────────────────────
- QUÉ PASÓ: Al mover el Dockerfile a la raíz, las rutas COPY (package.json,
  server.js) seguían apuntando al directorio actual (./) en vez de Backend/.
- IMPACTO: Segundo despliegue falló.
- CAUSA RAÍZ: Error del Senior al ejecutar la corrección sin verificar que
  las rutas COPY fueran coherentes con la nueva ubicación.
- PREVENCIÓN: Toda modificación de archivos de infraestructura debe incluir
  una revisión línea por línea de las rutas relativas.

ERROR #3: INSTALACIÓN DE PIP/YT-DLP VÍA PIP INESTABLE EN ALPINE
────────────────────────────────────────────────────────────────
- QUÉ PASÓ: La instalación original usaba `pip3 install yt-dlp` con el flag
  `--break-system-packages`. En Alpine Linux esto es frágil.
- IMPACTO: Potencial fallo silencioso en la instalación de yt-dlp.
- CAUSA RAÍZ: El Agente Backend copió un patrón genérico de instalación sin
  considerar las particularidades de Alpine Linux.
- PREVENCIÓN: Preferir siempre la descarga directa del binario oficial de
  yt-dlp desde GitHub Releases para entornos Alpine/Docker.

ERROR #4: FFMPEG NO ENCONTRADO — "spawn ffmpeg ENOENT"
──────────────────────────────────────────────────────
- QUÉ PASÓ: El servidor Node.js invocaba `spawn('ffmpeg', ...)` sin ruta
  absoluta. En el contenedor Alpine, ffmpeg estaba en /usr/bin/ffmpeg pero
  Node.js no siempre hereda el PATH correctamente.
- IMPACTO: La descarga de video fallaba con error 500 cada vez que el usuario
  hacía clic en "Descargar Ahora". El análisis funcionaba pero la descarga no.
- CAUSA RAÍZ: El Agente Backend asumió que el binario estaría accesible por
  nombre. El Senior aprobó sin verificar la resolución de PATH en Docker.
- PREVENCIÓN: SIEMPRE usar rutas absolutas o variables de entorno para binarios
  externos en Docker. Patrón estándar:
  `const FFMPEG_BIN = process.env.FFMPEG_PATH || '/usr/bin/ffmpeg';`

ERROR #5: FRONTEND NO SERVIDO DESDE RENDER — "Cannot GET /"
────────────────────────────────────────────────────────────
- QUÉ PASÓ: El servidor Express solo tenía endpoints API (/api/ping, /api/info,
  /api/download). No servía ningún archivo estático. El usuario esperaba ver la
  interfaz al abrir https://nexstream-pro.onrender.com pero solo veía un error.
- IMPACTO: Confusión total del Director. La arquitectura Cliente-Servidor estaba
  diseñada para desplegar Frontend y Backend por separado (Vercel + Render), pero
  esto no fue comunicado claramente al Director, quien esperaba una URL única.
- CAUSA RAÍZ DOBLE:
  a) El Senior no alineó expectativas con el Director sobre cómo acceder al producto.
  b) La arquitectura original (Frontend en Vercel, Backend en Render) era válida
     pero innecesariamente compleja para un MVP. Servir el HTML desde Express era
     la solución obvia y más simple.
- PREVENCIÓN: Para MVPs y proyectos de un solo día, SIEMPRE servir el frontend
  desde el mismo servidor backend usando express.static(). Separar frontend y
  backend solo cuando haya razones técnicas reales (CDN, SSR, etc.).

ERROR #6: MÉTODO DE DESCARGA INCOMPATIBLE — ERROR 404 AL DESCARGAR
──────────────────────────────────────────────────────────────────
- QUÉ PASÓ: El Frontend usaba un anchor tag dinámico con `target='_blank'` y
  `link.download` para iniciar la descarga. Esto generaba un 404 porque el
  navegador intentaba navegar a la ruta como si fuera una página HTML.
- IMPACTO: El botón "Descargar Ahora" no descargaba nada. El usuario veía
  "Listo" en el botón pero no recibía ningún archivo.
- CAUSA RAÍZ: El Agente Frontend implementó un patrón de descarga incorrecto.
  El Senior aprobó el código sin probarlo en un entorno real.
- PREVENCIÓN: Usar `window.open(downloadUrl, '_blank')` para descargas de
  archivos que provienen de un API endpoint. Evitar anchor tags dinámicos
  con target=_blank para este tipo de operaciones.

ERROR #7: API_BASE HARDCODEADO A LOCALHOST
──────────────────────────────────────────
- QUÉ PASÓ: El Frontend tenía `const API_BASE = 'http://localhost:3000'`
  hardcodeado. Esto hacía que en producción (Render) las llamadas API fueran
  a localhost en vez del servidor real.
- IMPACTO: Si el Senior no lo hubiera corregido, NADA habría funcionado en
  producción. Todas las llamadas API habrían fallado.
- CAUSA RAÍZ: El Agente Frontend dejó un placeholder de desarrollo sin
  implementar detección automática del entorno.
- PREVENCIÓN: Implementar SIEMPRE detección automática de entorno:
  `const API_BASE = window.location.protocol === 'file:' ? 'URL_RENDER' : '';`

ERROR #8: DESCARGA VÍA BLOB (ORIGINAL) — RIESGO DE COLAPSO EN MÓVILES
─────────────────────────────────────────────────────────────────────
- QUÉ PASÓ: La implementación original del Frontend usaba `await res.blob()`
  para descargar el video completo a la memoria del navegador antes de guardarlo.
- IMPACTO: En dispositivos móviles con poca RAM, videos de más de 50MB podrían
  causar un crash del navegador.
- CAUSA RAÍZ: El Agente Frontend usó un patrón estándar de fetch sin considerar
  el tamaño de los archivos multimedia.
- PREVENCIÓN: NUNCA usar blob() para archivos multimedia grandes. Siempre
  delegar la descarga al navegador nativamente.

NUEVAS REGLAS PARA EL SISTEMA MAPO
================================================================================

REGLA 1 — PRUEBA ANTES DE APROBAR:
El Agente Senior NO puede aprobar el trabajo de ningún agente sin evidencia
de al menos UNA prueba funcional exitosa (screenshot, log de terminal, o
resultado de endpoint).

REGLA 2 — DOCKERFILE EN RAÍZ:
Todo Dockerfile debe ubicarse en la raíz del repositorio. Las rutas COPY
deben usar prefijos de subcarpeta (ej: COPY Backend/server.js ./).

REGLA 3 — RUTAS ABSOLUTAS EN DOCKER:
Todo binario externo (ffmpeg, yt-dlp, python, etc.) debe invocarse con
ruta absoluta o variable de entorno dentro de contenedores Docker.

REGLA 4 — DETECCIÓN AUTOMÁTICA DE ENTORNO:
Todo frontend debe detectar automáticamente si está en desarrollo (localhost/
file://) o producción y ajustar sus URLs de API en consecuencia. Prohibido
dejar hardcodeado `localhost` en código que se desplegará.

REGLA 5 — MVP = MONOLITO:
Para proyectos de un solo día o MVPs, servir frontend y backend desde el
mismo servidor. No separar en múltiples servicios a menos que haya una
razón técnica justificada y aprobada por el Director.

REGLA 6 — AGENTE DEVOPS:
Para proyectos que involucren despliegue en la nube (Docker, PaaS, CI/CD),
se debe incorporar un Agente Especialista en Despliegue/Cloud (DevOps).
No es responsabilidad del Backend Developer gestionar la infraestructura
de deployment.

REGLA 7 — VERIFICACIÓN DE DESCARGA:
Todo mecanismo de descarga de archivos debe probarse end-to-end antes de
aprobarse. El patrón recomendado es window.open() para archivos servidos
desde endpoints API. Prohibido usar blob() para archivos multimedia grandes.
